# Desktop App ÔÇö macOS Shell

SwiftUI macOS app wrapping the Bristlenose React SPA in a WKWebView. Native project sidebar, native toolbar, native Settings window, web content in embedded mode. **Alpha ships a bundled, signed PyInstaller sidecar** that runs `bristlenose serve` inside the `.app`, distributed via internal TestFlight. The current code uses **launcher-style scaffolding** (the Swift app searches the user's `$PATH` and venv directories for an installed `bristlenose` CLI rather than carrying its own) so v0.2 native-shell work could iterate without paying the bundle-sign-distribute tax on every change. Track C in Sprint 2 replaces it with the bundled sidecar.

## Shipping architecture (alpha and beyond)

See `docs/design-desktop-python-runtime.md` for the canonical design. Summary:

- **Bundled, signed PyInstaller sidecar.** The Python `bristlenose serve` runtime ships inside `Bristlenose.app/Contents/Resources/bristlenose-sidecar/`. Every `.dylib`, `.so`, and framework inside is individually codesigned with Hardened Runtime. FFmpeg + ffprobe bundled alongside (trimmed codec set, ~25 MB). Whisper transcription model downloaded on first run to `~/Library/Application Support/Bristlenose/models/` ÔÇö keeps the `.app` under ~200 MB, matches Aiko/MacWhisper/Audio Hijack UX.
- **Why sidecar, not embedded interpreter.** Long-running pipeline operations (transcription, LLM calls) must be independently killable from Activity Monitor. An embedded Python interpreter takes the SwiftUI chrome down if Python hangs ÔÇö beach ball on the toolbar, force-quit the whole app. Separate process keeps that failure mode contained.
- **Single Python artefact.** Same `bristlenose serve` code path ships via Homebrew/Snap/pip for CLI users and via the `.app` for desktop users. No fork.
- **Apple-supported pattern.** Apple Developer Support recognises three App-Store-supported Python architectures (pure-Python wrapper via Briefcase, sidecar via IPC, embedded interpreter via PythonKit). Sidecar is documented and shippable.
- **App Sandbox on, Hardened Runtime on** for Release / TestFlight. Debug builds may relax either to speed iteration.

**Development history:** v0.1 (Feb 2026, in `desktop/v0.1-archive/`) proved the bundling pipeline end-to-end with a one-shot `bristlenose run` wizard. v0.2 (Mar 2026 onwards, this codebase) rebuilt the native shell ÔÇö NavigationSplitView workspace, WKWebView, native sidebar/menu/settings ÔÇö using the simpler launcher pattern so the native work could iterate without the bundle-sign-distribute tax on every change. Alpha recombines them: v0.2's native shell + v0.1's bundling pipeline, adapted to the long-lived `bristlenose serve` lifecycle.

### State ownership

`ServeManager` and `BridgeHandler` are `@StateObject` at the **App level** (not ContentView). This is required because:
- `.commands {}` needs `bridgeHandler` to dispatch tab switches
- `.onReceive(willTerminateNotification)` needs `serveManager` to kill zombies
- Both are passed to ContentView via `.environmentObject()`

ContentView uses `@EnvironmentObject` ÔÇö it does not own these objects.

### Tab enum and route mapping

`Tab.swift` defines the 5 toolbar tabs: `project`, `sessions`, `quotes`, `codebook`, `analysis`. Raw values match `TAB_ROUTES` keys in `frontend/src/shims/navigation.ts`.

`Tab.from(path:)` derives the active tab from a URL path using prefix matching. Order matters ÔÇö longest prefixes checked first so `/report/sessions/abc123` maps to `.sessions`, not `.project`. The project tab uses exact match (`== "/report/"`) to avoid swallowing all paths.

### Bridge communication

**Inbound** (web ÔåÆ native): `WKScriptMessageHandler` receives messages from `window.webkit.messageHandlers.navigation.postMessage(...)`. Types: `ready`, `route-change`, `editing-started`, `editing-ended`, `focus-change`, `undo-state`, `player-state`, `project-action`, `find-pasteboard-write`.

**Outbound** (native ÔåÆ web): `BridgeHandler` holds a `weak var webView: WKWebView?` (set in `WebView.makeNSView`). Five outbound methods:
- `goBack()` / `goForward()` ÔÇö delegates to `webView?.goBack()` / `.goForward()`
- `switchToTab(_ tab: Tab)` ÔÇö calls `callAsyncJavaScript("window.switchToTab(tab)", ...)`
- `menuAction(_ action: String, payload:)` ÔÇö calls `callAsyncJavaScript("window.__bristlenose.menuAction(action, payload)", ...)`. Single dispatch point for all ~89 menu actions (security rule 3 ÔÇö structured arguments, no string interpolation)
- `reloadWebView() -> Bool` ÔÇö `webView?.reloadFromOrigin()`; returns false when the webView is momentarily nil mid project-switch so the caller can retry. Swaps the stale status page for the freshly-served report after a run finishes ÔÇö see the report-auto-reload gotcha below

**`callAsyncJavaScript` has two overloads ÔÇö choose by whether you read the result.** The async form is `(_:arguments:in:contentWorld:) async throws -> Any?` (second label `contentWorld:`); the completion-handler form is `(_:arguments:in:in:completionHandler:) -> Void` (both frame and world use `in:`). A `try await` call that **binds** the return value MUST use `contentWorld:` ÔÇö writing `in: nil, in: .page` there silently resolves to the *completion* overload, whose `Void` result then won't bind (`conditional binding must have Optional type` ÔÇö the TypeParity `__typeParityCollect` build break fixed in `991635c3`). Content world `.page` is required either way because `window.switchToTab` / `window.__bristlenose.menuAction` are installed by page-level JS, not a `WKUserScript`.

**Known wrinkle (BridgeHandler):** `switchToTab` / `setLeftPanel` / `setLocale` / `menuAction` all write `try[?] await ÔÇª in: nil, in: .page` ÔÇö i.e. they `await`/`try` the *completion* overload. The JS still dispatches (fire-and-forget), but the `await`/`try` are no-ops and the `switchToTab` + `menuAction` `catch` blocks are **dead** (six `no 'async' operations occur within 'await'` + two `'catch' block is unreachable` warnings at build, on `BridgeHandler.swift:155/164/184/208/237/238`). Harmless today, but the inert error handling wants a dedicated fix ÔÇö either make them real `contentWorld:` async calls (keep the `catch`) or drop the dead `try/await/catch`. Don't copy this pattern into new bridge calls.

**Report auto-reload after a run finishes ÔÇö don't reach for `.id`.** When a run completes and the user *stays* on the project, the detail WebView is still on the serve's status page ("Nothing to see here, yet."). The serve re-imports the report on the `run_completed` terminus within ~1s and sets `last_run`, but the WebView never reloads itself. Three traps, all hit on the determinate-progress branch (`ContentView.scheduleReportReloadOnCompletion`):
- **`.id(project.id)`-bump recreation does not reliably reload.** SwiftUI may *reuse* the `NSViewRepresentable` rather than recreate it, running `updateNSView` ÔÇö whose `guard url != lastLoadedURL` short-circuits because the serve URL never changes. The bump silently no-ops. Reload directly via `bridgeHandler.reloadWebView()` (`reloadFromOrigin`, bypasses cache) instead.
- **`bridgeHandler.isReady` is NOT "the report is showing."** The status page never posts `ready`, and `WebView.didFinish` force-sets `isReady = true` 2s after *any* load ÔÇö so it stays true on stale content and never resets without a project switch. Don't gate render-state logic on it.
- **Gate the trigger on `analysing ÔåÆ ready`, not `!ready ÔåÆ ready`.** The launch-time `nil ÔåÆ ready` disk read of an already-finished project otherwise fires a spurious reload (and bails because the serve is still booting). Project *switches* don't need this path at all ÔÇö the WebView is recreated fresh, and a serve-port change reloads via `updateNSView`; only the no-switch completion does.

### Toolbar morphing

**Principle: menus dim, toolbars morph.** The menu bar greys out unavailable items (HIG ÔÇö discoverability). The toolbar shows/hides items per tab (Photos/Notes pattern ÔÇö no greyed-out button graveyard).

**Universal items** (always present):
- **Search** (magnifying glass) ÔÇö active on Quotes tab, dimmed elsewhere as "coming soon"
- **Export** (share icon) ÔÇö dropdown `Menu` whose contents change per tab. Always has "Export Report..." first. Quotes tab adds "Export Quotes as CSV"

**Per-tab contextual items** (appear/disappear):
- **Quotes/Codebook/Analysis**: `ControlGroup` pill with two buttons ÔÇö left (`sidebar.left`) toggles native project sidebar via `NSSplitViewController.toggleSidebar`, right (`list.bullet`) toggles web navigation sidebar (sections/themes, codebooks, signals) via `bridgeHandler.menuAction("toggleLeftPanel")`
- **Quotes**: Tag sidebar toggle (`sidebar.right` icon)
- **Analysis**: Heatmap inspector toggle (`square.grid.2x2` icon)
- **Project/Sessions**: no extra items

**The toolbar is the *only* surface for the web nav/tag sidebars in embedded mode.** The SPA hides its own icon rails + close-├ù when `isEmbedded()` (`.layout.embedded` in `bristlenose/theme/organisms/sidebar.css`) ÔÇö that web chrome is a non-Mac idiom and the two 36px rails wasted width the WKWebView (zero-inset, fills the detail pane) gives back to content. So the `toggleLeftPanel` / `toggleRightPanel` ControlGroup buttons + `[`/`]` keys are the user's only way to open/close them; there's no in-pane affordance to fall back on. Overlay/hover-peek mode is also gone in embedded (it was rail-triggered) ÔÇö the web TOC is closedÔåöpush only. If you ever remove these toolbar toggles, the web sidebars become unreachable in the app. See `docs/design-sidebar.md` ┬º Desktop embedded mode.

`ExportMenuButton` is a `View` struct in `ContentView.swift` that observes `bridgeHandler.activeTab` to render the correct menu items.

### Menu bar

`MenuCommands.swift` ÔÇö full native menu bar with 10 menus and ~89 items.

**`View`-inside-`Commands` pattern:** `@ObservedObject` is unreliable directly in `Commands.body` (observation doesn't trigger re-evaluation). `MenuCommands` holds a plain `let bridgeHandler`. Each menu section is a `View` struct (`QuotesMenuContent`, `VideoMenuContent`, etc.) that owns `@ObservedObject var bridgeHandler` ÔÇö views inside `CommandMenu`/`CommandGroup` follow normal SwiftUI view lifecycle.

**Contextual dimming** ÔÇö menus dim unavailable items (never hide):
- Quotes menu: all items dim when not on Quotes tab. Star/Hide/Tag additionally require `focusedQuoteId != nil`
- Video menu: all items dim when `!hasPlayer`. Play/Pause label swaps
- Codes menu: group/code operations dim when not on codebook/quotes tab. Browse/Import always enabled
- Edit > Undo/Redo: hidden when `isEditing` (lets WKWebView handle character-level undo)
- View > panels: dim based on active tab

**Responder chain rules:**
- Do NOT touch `.pasteboard` ÔÇö Cut/Copy/Paste handled by WKWebView responder chain
- Undo/Redo hidden during `isEditing` to let Cmd+Z fall through to WKWebView
- Cmd+F routes to web search bar (not native WKWebView find bar)
- Settings Cmd+, comes from the `Settings {}` scene automatically ÔÇö no custom menu item

**No bare-key menu shortcuts** ÔÇö `s`, `h`, `[`, `]`, `m`, `?`, arrows work only in WKWebView focus. Menu items for these actions have no keyboard shortcut shown. Help menu points to `?` for the full shortcut reference.

### KVO for back/forward

`WebView.Coordinator` stores two `NSKeyValueObservation` properties observing `webView.canGoBack` and `webView.canGoForward`. KVO callbacks dispatch to `bridgeHandler.canGoBack`/`.canGoForward` via `Task { @MainActor in }`. Observations are auto-invalidated when the Coordinator deallocates (triggered by `.id(project.id)` on project switch).

### Zombie process cleanup

**The host doesn't enumerate orphans ÔÇö it can't, by sandbox design.** App Sandbox blocks `proc_listpids`, `lsof`, `ps`, and friends. Instead, the orphan cleans itself: the sidecar (Python) installs a parent-death watcher that polls `os.getppid()` every 2s; on detecting reparenting (PID changes ÔÇö typically to launchd/PID 1), it SIGTERMs itself for uvicorn's graceful-shutdown path. See `bristlenose/server/lifecycle.py`.

**Read process metadata via libproc, never via subprocess.** Four swaps now in the family ÔÇö Swift `proc_pidpath` (zombie-cleanup PID verification, commit `38808fe`), Swift `proc_listpids`+`proc_pidfdinfo` (TCP-LISTEN socket walk, commit `5471b35`), Python `proc_pidinfo(PROC_PIDTBSDINFO)` for process start time in `bristlenose/run_lifecycle.py:_ps_start_time` (May 2026), and Swift `proc_pidinfo(PROC_PIDTBSDINFO)` for the PID-liveness check in `EventLogReader.procStartTime` (Jun 2026). All `/bin/ps` and `/usr/sbin/lsof` execs are sandbox-blocked at startup, not at first call ÔÇö adding a fifth subprocess-shaped helper will crash the bundled sidecar before it serves a byte. If a fifth need arises, that's the trigger to extract `bristlenose/utils/libproc.py` (Rule of Three, long met).

**The Swift/Python `start_time` token is a cross-language equality contract ÔÇö keep both sides byte-identical.** Python writes `run.pid`'s `start_time` and Swift (`EventLogReader.pythonOwnedRunIsAlive`) reads it back to confirm a run is still live; they compare for *equality*, never parse. On macOS both must emit `proc_pidinfo(PROC_PIDTBSDINFO)`'s `"<pbi_start_tvsec>.<pbi_start_tvusec>"`. The original Swift side read `/bin/ps -o lstart=` (human date string) ÔÇö which both mismatched Python's libproc format on every real run *and* was sandbox-blocked ÔÇö so `.running` detection silently never fired in the sandboxed build. A self-consistent test (Swift writes ps-format, Swift reads ps-format) masked it for months. If you touch either `_ps_start_time` (Python) or `procStartTime` (Swift), change both, and make the test build its PID file via the production query rather than hand-rolling the format.

The host signals "I am the parent that owns this lifecycle" by setting the `_BRISTLENOSE_HOSTED_BY_DESKTOP=1` env var when spawning the sidecar. CLI users don't get the watcher ÔÇö they may legitimately `nohup` and outlive their starting shell.

**Subprocess environment ÔÇö build it with `BristlenoseShared.childEnvironment(for: mode)`, never hand-roll.** Every desktop-spawned `bristlenose` subprocess needs five env concerns, and assembling them by hand at each call site is exactly how they drift. `childEnvironment` is the single source of truth, folding in: (1) the minimal var allowlist + UserDefaults preferences (`buildChildEnvironment` ÔåÆ `overlayPreferences`); (2) `_BRISTLENOSE_HOSTED_BY_DESKTOP=1` (parent-death handshake); (3) TLS cert paths (`sslEnvironment` ÔÇö PyInstaller's bundled OpenSSL points at Homebrew paths blocked by App Sandbox, so the 4 SSL vars redirect Python to the certifi `cacert.pem` inside the `.app`); (4) bundled FFmpeg/ffprobe paths (`bundledBinaryEnvironment`); (5) the active provider's API key (`overlayAPIKeys` ÔÇö read from Keychain by the Swift host, since Python can't touch Keychain under sandbox). Used in `PipelineRunner.start()` and `ServeManager.start()`. **Any new spawn site calls `childEnvironment`, full stop ÔÇö don't re-derive the env block.** History: the run path used to hand-roll the block and silently dropped the API-key overlay, which broke `bristlenose run` the moment sandbox was enabled (Python got no key). The factory exists so that class of bug can't recur. The SSL/ffmpeg/key sinks are also why subprocess stdout is redacted with `BristlenoseShared.redactKeys(in:)` before it reaches the diagnostic-popover buffer at both spawn sites ÔÇö a Python-side key echo must not ride "Copy error details" out to a pasted bug report.

**Desktop type system ÔÇö activated here (this was the "critical gap").** `childEnvironment` now sets `env["BRISTLENOSE_PLATFORM"] = "desktop"`, so the server emits `data-platform="desktop"` and the SF Pro + Apple-HIG type scale in `tokens-desktop.css` renders (it shipped dark before ÔÇö the cascade was unplugged at this env seam). It also pins `env["BRISTLENOSE_COLOR_THEME"] = "default"` so app.py doesn't default desktop ÔåÆ the Edo palette (deferred, not for TestFlight ÔÇö flip to `"edo"` or make it a preference when those colours land). **Seeing it still needs a sidecar rebuild + clean build** (a Cmd+R reuses the stale bundle); the Dev Sidecar scheme sidesteps the rebuild. The **Appearance Ôû© Typography** picker (`AppearanceSettingsView`, `@AppStorage("typography")`, default `"sf"`) lets the user opt back to Inter: `overlayPreferences` injects `BRISTLENOSE_TYPOGRAPHY="inter"` ÔåÆ app.py emits `data-typography="inter"` ÔåÆ `tokens-desktop.css`'s `:not([data-typography="inter"])` gate falls back to `:root` (Inter). The change lands on the next serve start (the picker posts `.bristlenosePrefsChanged` ÔåÆ restart, same mechanism as appearance/language). See `docs/design-native-typography-grid.md`.

Two layers in total:

1. **Clean quit**: `.onReceive(NSApplication.willTerminateNotification)` on the root View calls `serveManager.stop()` (SIGINT). Sidecar shuts down within milliseconds.
2. **Crash recovery (host SIGKILLed by Xcode stop button, force-quit, OOM)**: sidecar's parent-death watcher detects reparenting within 2s and SIGTERMs itself. The next host launch uses `bind(0)`, so even if the orphan is mid-shutdown, the new sidecar gets a different kernel-assigned port. Result: silent recovery, no UI surface.

**Why bind(0) instead of a deterministic per-project port:** the desktop host doesn't need port stability across launches (WKWebView gets injected with whatever URL the sidecar reports, every time). CLI users keep the deterministic 8150 default for bookmark-and-reload. Decoupling the two contracts removed ~150 lines of djb2-port-walker + libproc-cleanup scaffolding (Apr 2026, A6 redesign).

**Side effects of the redesign:**
- Sidecar prints `Report: http://127.0.0.1:NNNN/report/` *after* socket bind (kernel-assigned port). `ServeManager.handleLine(_:)` parses the port from this URL via `reportPortRegex`.
- Every sidecar exit emits one structured INFO line: `sidecar_exit reason=<r> uptime_sec=<u> original_ppid=<o> current_ppid=<c> [exc=<e>]`. Reasons: `parent_death`, `sigterm`, `sigint`, `uncaught_exception`, `normal_shutdown`. Greppable from `<output_dir>/.bristlenose/bristlenose.log` ÔÇö first stop when investigating "why did the sidecar die?".
- The sidecar's watcher works on **its own state only** (own getppid, own SIGTERM). No process enumeration. Sandbox-clean.

`bristlenose run` subprocesses (the pipeline) have a separate PID-file scan/attach lifecycle managed by `PipelineRunner`; see `docs/design-subprocess-lifecycle.md`. The serve sidecar's watcher does not cascade to run subprocesses.

**Cancelling a `bristlenose run` subprocess needs signal escalation, not just SIGINT.** Whisper / torch / ctranslate2 hold the GIL during long C calls (model load can take 30ÔÇô60s). Python signal handlers only run between bytecodes ÔÇö SIGINT and SIGTERM both sit queued during the wedge, so a Stop click sees no effect for tens of seconds. `PipelineRunner.scheduleOrphanCancelEscalation` does SIGINT ÔåÆ SIGTERM at +5s ÔåÆ SIGKILL at +8s; bails at any step if the PID is dead or `attachedOrphanPIDs[id]` no longer matches. SIGKILL bypasses Python entirely so it always wins. Owned-process cancel still uses `proc.interrupt()` (SIGINT only) ÔÇö same wedge risk, escalation TODO.

**`applyScanResult` won't overwrite `.running`.** Passive sidebar manifest scans must not clobber active runs, so the function returns early on `.running`/`.queued`/`.failed`. `handleOrphanExit` IS the "run is over" signal ÔÇö it must explicitly transition `state[projectID] = .idle` itself before scheduling the manifest re-read, or the pill stays "Running" forever after the subprocess dies.

**`PipelineProgress.isStopping` is the immediate-ack contract.** Set unconditionally at the top of `cancel()` for both owned and orphan paths so the toolbar pill, popover, and sidebar row all flip to "StoppingÔÇª" before the kill propagates. Without this, the user mashes Stop while signals play out (1ÔÇô8s).

**Sandbox implication for alpha:** `/usr/sbin/lsof` exec is blocked by App Sandbox. In the shipped build this cleanup path will silently fail. The user-visible consequence is a "port in use" error on the next launch after a crash, which should be handled with a clear restart prompt. Alternatively replace with a Swift-native TCP connect sweep across the port range. Decision deferred to C1.

## Security rules

1. **Navigation restriction** ÔÇö `decidePolicyFor` allows only `127.0.0.1` and `about:`. External URLs open in system browser via `NSWorkspace.shared.open()`.
2. **Bridge origin validation** ÔÇö every `WKScriptMessageHandler` callback checks `message.frameInfo.request.url?.host == "127.0.0.1"`.
3. **No string interpolation into JavaScript** ÔÇö use `callAsyncJavaScript(_:arguments:in:in:)` for nativeÔåÆweb calls. Never concatenate user data into `evaluateJavaScript` strings. A project named `'; alert(1); '` must not become code execution.
4. **Ephemeral storage** ÔÇö each project gets `WKWebsiteDataStore.nonPersistent()` to prevent cross-project cookie/sessionStorage leakage.
5. **Settings interception** ÔÇö `project-action: open-settings` opens the native Settings scene, not the web modal.

## WKWebView cross-view messaging

Multiple WKWebViews can communicate via **BroadcastChannel** if they share the same `WKWebsiteDataStore` instance. Validated in spike (28 Mar 2026, macOS 26.1). Full design: `docs/design-wkwebview-messaging.md`.

**Critical rule:** `.nonPersistent()` creates a new ephemeral partition on every call. Two views that each call `.nonPersistent()` independently are fully isolated. To enable BroadcastChannel between views, both must use the **same instance** via a singleton:

```swift
class SharedConfigStore {
    static let shared = SharedConfigStore()
    let dataStore: WKWebsiteDataStore = .nonPersistent()
    let processPool: WKProcessPool = WKProcessPool()
}

// In both WebView configs:
config.websiteDataStore = SharedConfigStore.shared.dataStore
```

Process pool sharing is optional ÔÇö separate pools do not break BroadcastChannel. Only the data store instance matters.

**Channel naming:** `bristlenose-{purpose}-{projectId}` (e.g. `bristlenose-tags-1`). First consumer: Tag Inspector panel (`docs/design-native-inspector.md`).

**Each WKWebView needs its own `WKWebViewConfiguration`** ÔÇö don't share a config object between views (each has its own `userContentController` for message handlers). Share the data store and pool *via* the config, not the config itself.

## Port allocation

**Desktop:** the host passes `--port 0` to the sidecar; the kernel assigns a free port via `bind(0)`. `ServeManager.handleLine(_:)` parses the port from the sidecar's `Report: http://127.0.0.1:NNNN/report/` stdout line (printed *after* socket bind). Every host launch gets a fresh port ÔÇö no contention with slow-dying orphans, no port-walking retry loop, no need for a deterministic-port table.

**CLI:** `bristlenose serve` defaults to port 8150 for backwards compatibility (CLI users bookmark `http://127.0.0.1:8150/report/` and reload after re-running the pipeline). Pass `--port 0` to opt into kernel assignment if you need it.

The deterministic djb2-per-project allocation was removed in Apr 2026 (A6 redesign) ÔÇö it only mattered for orphan-cleanup, which the sidecar now handles itself.

**WKWebView right-click ÔåÆ Copy Link** yields a single-session URL (the port changes next launch). Fine ÔÇö Bristlenose is not a browser; users don't bookmark internal URLs.

A **stable team identity** keeps Keychain access prompt-free across rebuilds ÔÇö but the *mechanism* changed with the 2 Jun data-protection-keychain migration (`8b2ef51`), which **supersedes** the old "Always Allow ACL bound to the binary hash" framing. `KeychainHelper` now uses the **data-protection keychain** (`kSecUseDataProtectionKeychain`) with a team-scoped `keychain-access-groups` entitlement (`$(AppIdentifierPrefix)app.bristlenose`); the DP keychain validates by **Team ID, not binary hash** (Apple TN3137; cf. steipete/CodexBar #585), so any team-signed build reads its own provider keys **silently and across rebuilds ÔÇö no per-key "Always Allow" prompt at all**. The one path that still re-prompts is **ad-hoc** signing (`None` + `Sign to Run Locally`): empty `$(AppIdentifierPrefix)` ÔåÆ access group breaks ÔåÆ fallback to the *legacy* file-based keychain, which binds grants to the binary hash and re-prompts every build. So the stable `Z56GZVA2QB` team matters for keeping the **access-group entitlement valid**, not for "ACL persistence." Contributors without that team configured will see "No signing certificate found for team Z56GZVA2QB" on Debug build ÔÇö fallback is Xcode ÔåÆ Bristlenose target ÔåÆ Signing & Capabilities ÔåÆ Debug ÔåÆ set Team to your own (or None + Sign to Run Locally). **Do not commit that local change.** Release config is independent (Apple Distribution + Mac App Store provisioning profile, unaffected by Debug signing choice). **First Cmd+R after switching identity fires two one-time non-Keychain prompts** ÔÇö (1) `codesign wants to access key "Apple Development: ÔÇª"` (signing-key Always Allow ÔÇö that's the *signing* private key in the login keychain, accessed by the build tool, unrelated to our provider keys), (2) `"Bristlenose" differs from previously opened versions` (LaunchServices, Open Anyway). _Superseded (pre-2-Jun, file-based keychain): a third "Keychain ACL-extension prompt per stored provider key" used to fire here (Verified 16 May 2026: macOS extended the existing item's ACL rather than orphaning it). The data-protection-keychain migration removed it for team-signed builds ÔÇö own-access-group reads no longer prompt._ Second Cmd+R is silent ÔÇö that's the test.

## Build

### Dev build (today)

```bash
# CLI build (uses xcodebuild)
cd desktop/Bristlenose
xcodebuild build -scheme Bristlenose -configuration Debug -destination "platform=macOS"

# Or open in Xcode and Cmd+R
open desktop/Bristlenose/Bristlenose.xcodeproj
```

The Xcode project uses `PBXFileSystemSynchronizedRootGroup` ÔÇö Swift files added to `desktop/Bristlenose/Bristlenose/` are auto-discovered. No need to manually add them to the project.

**The FIRST `xcodebuild` in a fresh worktree fails with `cannot find 'GeneratedBuildInfo' in scope` ÔÇö re-run, it succeeds.** `GeneratedBuildInfo.swift` is gitignored (doesn't follow worktrees) and is written by the pre-compile "Run Script" build phase (`scripts/generate-build-info.sh`). On the very first build of a fresh worktree the file doesn't exist yet, and the `PBXFileSystemSynchronizedRootGroup` snapshots the Compile-Sources list *before* the script creates it ÔÇö so that build can't see the just-generated file and errors. The file is now on disk, so the **second** build discovers it and compiles clean. Not a real error; don't chase it. (Same class as the gitignored-binaries-don't-follow-worktrees gotcha, but for a generated source file rather than ffmpeg/models.)

**Auto-sync only covers files inside `desktop/Bristlenose/Bristlenose/`.** Shared data dirs at the repo root (e.g. `bristlenose/locales/`) won't ship in the bundle unless an explicit Copy Bundle Resources phase or shell-script copy puts them there. Without that, `Bundle.main` lookups return nil at runtime ÔÇö symptom is chrome strings rendering as raw i18n keys (e.g. `desktop.settingsTabs.llm` instead of "LLM"). The "Copy Sidecar Resources" shell-script phase in `Bristlenose.xcodeproj/project.pbxproj` is the established pattern for this ÔÇö it `rsync`s the sidecar, ffmpeg, models, and locales (added Apr 2026 after locales were missing from the bundle on `main`). Add new shared dirs to that phase rather than inventing new ones.

**Swift i18n sweeps: grep helper-function `return "ÔÇª"` + `Button("ÔÇª")` / `.help("ÔÇª")`, not just `Text("ÔÇª")`.** Hardcoded user-facing strings hide in computed vars and static funcs that `return "ÔÇª"` (e.g. `PipelineActivityItem.headlineStatus`, `pillHelp`, `humanCategoryLabel`) and in `Button("ÔÇª")` / `.help("ÔÇª")` / `.accessibilityLabel("ÔÇª")`. A `Text("ÔÇª")`-only grep undercounts and ships a half-localised view (popover header English, body translated). Grep `return "[A-Z]`, `Button("`, `.help("`, `.accessibilityLabel("` too. **Watch static funcs shared with English-only surfaces:** `humanCategoryLabel` is the single source for the UI label AND the English-only `formatDiagnosticPlaintext` copy-payload ÔÇö translating it in place breaks the plaintext; it needs a UI-vs-plaintext split (cz-branch i18n review Finding 13, deferred).

## Dev workflow: three modes via Xcode scheme variants

After Track C C1 lands, three shared Xcode schemes map to three sidecar resolution modes. Pick the scheme in the Run-button dropdown next to Cmd+R. Each scheme pre-sets the appropriate env vars; Debug-only `#if DEBUG` guards mean Release builds physically exclude these reads and always use the bundled path.

| Scheme | Env vars it sets | Mode | Purpose | Failure mode |
|---|---|---|---|---|
| **Bristlenose** (default) | _(none)_ | `bundled` | What TestFlight ships. Spawns the sidecar at `Bundle.main.resourceURL/bristlenose-sidecar/bristlenose-sidecar`. Exercises the full shipping flow end-to-end. | Debug + bundle missing ÔåÆ `.failed` state + SwiftUI error card with "set `BRISTLENOSE_DEV_SIDECAR_PATH` or run `desktop/scripts/build-sidecar.sh`". Release + bundle missing ÔåÆ `.failed` + "installation corrupted" + Reveal in Finder. |
| **Bristlenose (External Server)** | `BRISTLENOSE_DEV_EXTERNAL_PORT=8150` | `external` | Fast CSS/React iteration. Start `bristlenose serve --dev trial-runs/project-ikea` in a terminal, then Cmd+R. App connects to the external server ÔÇö no subprocess. Live CSS endpoint re-reads on every request; Vite HMR pushes React changes. | App can't reach `127.0.0.1:<port>` ÔåÆ standard connection error + retry. Terminal server missing ÔåÆ same. |
| **Bristlenose (Dev Sidecar)** | `BRISTLENOSE_DEV_SIDECAR_PATH=/Users/cassio/Code/bristlenose/.venv/bin/bristlenose` | `devSidecar` | Swift spawns the exact binary you point at ÔÇö usually your dev venv. **Both env vars set simultaneously** ÔåÆ `ServeManager.init()` fails with "Both dev env vars set ÔÇö pick one." Catches configuration mistakes at launch rather than on first project click.

**Mode logging.** `ServeManager` uses `Logger(subsystem: "app.bristlenose", category: "serve")` and emits one line at startup: `Mode: external-server, port=8150` / `Mode: dev-sidecar, path=/Users/.../bristlenose` / `Mode: bundled, path=/Applications/.../bristlenose-sidecar`. Shows up in Xcode console, Console.app, and `log stream --predicate 'subsystem == "app.bristlenose"'` in Terminal. Release builds get automatic privacy redaction.

**Swap schemes via the dropdown** next to the Run button in Xcode. To set your own env-var value inside a scheme: Product ÔåÆ Scheme ÔåÆ Edit Scheme ÔåÆ Run ÔåÆ Arguments ÔåÆ Environment Variables. For personal paths (e.g. a non-default venv location), edit the scheme's env-var value directly ÔÇö the schemes are checked into `xcshareddata/xcschemes/` so the default values are shared, but personal overrides stay local until you commit them.

**Linux / Windows contributors** never touch this ÔÇö these env vars are read by Swift, not Python. If you're working on the Python sidecar or React frontend, run `bristlenose serve` directly from your terminal and use any browser; the macOS shell is out of scope.

See `docs/design-modularity.md` "External dev server" glossary entry for the implementation spec. Underlying resolver is `SidecarMode.resolve(externalPortRaw:sidecarPathRaw:bundleResourceURL:fileManager:)` ÔÇö a pure function (takes the two env-var raw strings as typed `Optional<String>` rather than an env dictionary, so the string literals live only inside the caller's `#if DEBUG` block and are absent from the Release Mach-O). Unit-tested in `BristlenoseTests/SidecarModeTests.swift` (orphan target today ÔÇö see test-target note in this file).

## Gotchas

- **The desktop failure popover/classifier can mislabel ÔÇö read the captured stderr, not the category.** `PipelineRunner.categoriseFailure` regex-matches the whole stderr *tail*, which always includes the `run`-startup doctor preflight block (`Ô£ô Transcription mlx-whisper`, `ÔÇö Whisper model ÔÇª not cached`). A broad `whisper` / `(speech )?model` pattern matches that *benign* preflight, not the real error ÔÇö a provider/model-mismatch **LLM 404** (`not_found_error ÔÇª model: gemini-2.5-flash`) was mislabelled "Transcription failed ÔÇö the speech model didn't load" for an entire debugging session (5 Jun 2026). **When a desktop run-failure label looks wrong, read the actual error** ÔÇö the diagnostic capture writes the redacted tail to `<project>/.bristlenose/last-run-failure.log` (added on `desktop-provider-resolution`), or read the run subprocess stderr; don't trust the popover category ("Detailed cause not captured" means the structured cause was empty and the category is a guess). Classifier rule when touching `categoriseFailure`: check specific LLM-error markers (`provider says`, `not_found_error`) BEFORE the whisper check, and require real whisper error signatures (`PackageInstallError` / `snapshot_download` / `metallib` / `libjaccl`), never bare `model` / `whisper` (the preflight always prints them). Full root cause + fixes: `docs/private/handoffs/desktop-provider-resolution.md`.
- **`OSError: could not get source code` at sidecar import time ÔåÆ typeguard 4.x + PyInstaller + a package whose `@typechecked` decorator runs at module-load.** Symptom: bundled sidecar crashes during `from bristlenose.cli import app` with the chain `typeguard/_decorators.py:70 instrument() ÔåÆ inspect.py:1096 findsource() ÔåÆ OSError`. Cause: typeguard 4.x rewrites decorated functions' ASTs via `inspect.getsource()` at import time. PyInstaller's PYZ archive holds `.pyc` only ÔÇö no `.py` source ÔÇö so the rewrite fails. **Fix:** add the offending package to the spec's `module_collection_mode` dict with value `'pyz+py'` so PyInstaller bundles both bytecode and source. Example in `desktop/bristlenose-sidecar.spec`: `module_collection_mode={'inflect': 'pyz+py'}`. The class of bug recurs whenever a transitive dep adopts typeguard / beartype / `@validate_call` at module level ÔÇö audit on every fresh `pip install` of `.venv-sidecar/`. Long-term protection: lockfile + spec compat registry in `docs/private/handoffs/sidecar-dep-lock.md`. Hit 16 May 2026 during the multi-project-folder-watcher acceptance walk.
- **SQLite `?immutable=1` URI parameter is load-bearing for sandboxed read-only opens of WAL-mode DBs.** Symptom: `sqlite3_prepare_v2` returns "unable to open database file" with `os_unix.c:51044 (2) open(/path/to/db-wal) - No such file or directory` in the os_log. Cause: SQLite WAL journal-mode databases require the reader to access (and sometimes create) `*-wal` and `*-shm` companion files. Under macOS Sandbox, a `SQLITE_OPEN_READONLY` connection can't write the dir to create the WAL companion ÔÇö it errors at prepare time. **Fix:** open via URI form with `?immutable=1`: `sqlite3_open_v2("file:\(path)?immutable=1", &db, SQLITE_OPEN_READONLY | SQLITE_OPEN_NOMUTEX | SQLITE_OPEN_URI, nil)`. This tells SQLite "this DB will not change during my session" ÔÇö skip the WAL/SHM dance entirely. Trade-off: snapshot read; writes mid-scan return prior state. See `SourceFilesReader.swift` for the canonical use. Hit 16 May 2026 on `bristlenose.db` reads from a sandboxed `ProjectFolderWatcher`. **Corollary (the count-blank-after-run bug, `1e1d608`): `immutable=1` can't see WAL-resident commits, so the *writer* must checkpoint or those rows are invisible.** The `sessions` count is written by the *serve importer* (`bristlenose/server/importer.py`), NOT the pipeline run, on a long-lived WAL connection that doesn't auto-checkpoint small imports ÔÇö so a just-imported count stayed in the `-wal`, the `immutable=1` sidebar read saw the stale main file (`nil`/old count), and it never self-corrected because the watcher's NSFilePresenter scope excludes `bristlenose-output/` (a DB-only change fires no rescan). The old "acceptable for watcher polling" framing was wrong for count data ÔÇö it doesn't change via source-file events, so "the next callback re-scans" never fires. Two-part fix: (1) the importer PASSIVE-checkpoints after every commit (`_checkpoint_wal`, see `bristlenose/server/CLAUDE.md`) so immutable readers see the rows; (2) `ContentView.scheduleCountRescan` rides out the serve's ~1s re-import with a few spaced `ProjectFolderWatcher.refresh()` rescans on the analysingÔåÆterminal transition (the count lands ~1s+ *after* the pipeline-exit signal that drives the rescan). **Caught only by the `silent-failure-hunter` review agent** ÔÇö four other agents + the plan all assumed the pipeline writes the count. When a desktop `immutable=1` read shows stale data, first check whether a *different process* (the serve) wrote it into an un-checkpointed WAL.
- **AppleDouble `._*` files break Python scanners that read every file as UTF-8.** Symptom: `bristlenose serve` crashes on a project copied to ExFAT/FAT32/SMB volumes with `UnicodeDecodeError: 'utf-8' codec can't decode byte 0xb0` reading `._<transcript>.txt`. Cause: macOS Finder writes AppleDouble metadata sidecars (resource-fork + xattrs) as `._<filename>` files when copying to non-APFS filesystems. They're binary metadata, not user content. Python scanners that iterate `*.txt` pick them up. **Fix:** filter `._*` at every scan site ÔÇö `bristlenose.utils.audio` ffprobe loop, `load_transcripts_from_dir`, source-file collection in `s01_ingest`. See `feedback_mac_filesystem_detritus.md` memory for the full list of macOS detritus filenames to skip. Hit 16 May 2026 during a Step 6 SD-card eject test; user workaround was `dot_clean /Volumes/X`. Chip filed for the durable fix.
- **Apple HIG pages are JavaScript-rendered ÔÇö `curl` / `WebFetch` returns only the static shell.** Loading `https://developer.apple.com/design/human-interface-guidelines/<slug>` from a non-browser client gets the HTML wrapper with `<title>Apple Developer Documentation</title>` and no body content. The real content is fetched client-side from the DocC JSON API. **Confirmed working endpoint** (probed 15 May 2026): `https://developer.apple.com/tutorials/data/design/human-interface-guidelines/{slug}.json` ÔÇö slug matches the URL path segment (`sidebars`, `color`, `typography`, etc.). Returns full DocC JSON: `primaryContentSections[0].content[]` carries the actual body (paragraph/heading/aside/list nodes); `references{}` resolves inline link identifiers; `metadata.customMetadata.alert-date` is the change-log freshness marker. The JSON also covers macOS-26-Tahoe-era updates the training cut may have missed. When citing HIG conventions in design discussions, **fetch the JSON and quote the actual section** rather than recalling from training ÔÇö my training was demonstrably fuzzy on typography roles, semantic colour usage, and sidebar conventions specifically (15 May 2026 session). - **`ServeManager` serve-lifecycle ownership is guarded by the `generation` counter ÔÇö and `try? await Task.sleep` swallows `CancellationError`, so a bare `Task.cancel()` is toothless.** Every serve-lifecycle driver (`start`, `shutdown(timeout:)`, `switchProject(to:)`, the readiness + idle-timeout polls) is async and can be superseded mid-`await` by a newer `start()` ÔÇö the common case is rapid project switches now that the cancel-on-switch modal is gone (`background-runs-view-switch`, A1). The supersession contract: `start()` bumps `generation` (`ServeManager.swift:139`); any path that spans an `await` captures `let myGeneration = generation` at entry and re-checks `guard generation == myGeneration` (or `!Task.isCancelled`) **after** each await, **before** writing terminal state (`process`, `state`, `readTask`) ÔÇö see `shutdown()` `:306`/`:360` and `switchProject()` `:420`. The trap: the poll loops use `try? await Task.sleep(...)`, which **catches and discards `CancellationError`** ÔÇö so storing a `switchTask` in `ContentView` and calling `.cancel()` on the next switch does **not** actually stop the in-flight teardown; only the explicit `Task.isCancelled`/`generation` checks do (this was the original Step-2.5 no-op trap ÔÇö "store + cancel" looked like a fix but was inert). Without the post-await guard, a slow `shutdown()` that loses the race clobbers the winning `start()`'s state (Finding 18, fixed `717afa9`). **When adding a new serve-lifecycle path (e.g. any `restartIfRunning`-style driver), reuse the single `generation` token ÔÇö do NOT add a second epoch counter or scatter ad-hoc `isCancelled` checks** (per `feedback_no_legacy_naming_in_new_code` and parsimony ÔÇö one ownership token, not many). **A2's warm-sidecar pool (below) is now implemented and did exactly this** ÔÇö the synchronous re-point bumps `generation` so a late cold-start readiness completion can't clobber it, and it actually *retired* the termination handler's epoch capture in favour of `ObjectIdentifier` identity routing (one identity check subsumes the fronted-vs-parked routing and the superseded-process no-op).
- **Warm-sidecar pool (Phase A2) ÔÇö `ServeManager` keeps ONE parked sidecar, and the detail-pane WebView must be keyed on `project.id` + serve port.** Switching the selected project used to teardown+restart the serve sidecar (slow boot, rapid-switch crash race). Single slot, not a dict+LRU pool (only the AÔåöB repro is observed; see `docs/private/reviews/warm-sidecar-pool.md` F15). **The load-bearing gotcha: the detail-pane WebView is keyed `.id("\(project.id)-\(port)")`, NOT `.id(project.id)`.** A warm re-point keeps the same `project.id` but a *different* port; keying on id alone would NOT re-mount, so `WebView.updateNSView` would reload the new port while reusing the *previous* sidecar's injected auth token (token is injected in `makeNSView` only) ÔåÆ silent 401s ÔåÆ blank report that looks like success (the BUG-3 class). The reliability win is total for warm hits; a 3rd-distinct-project cold start retains the existing (A1-guarded) boot race. Audit it manually on every state addition. **And audit `PipelineState.keepsMachineAwake`** (next gotcha) ÔÇö a new "work is executing" case must return `true` there or a run in that state will let the Mac idle-sleep.
- **Idle-sleep is held off during a run via `RunSleepAssertion` (`ProcessInfo.beginActivity(.userInitiated)`), driven off `state.didSet`.** A long unattended analysis on AC power used to freeze at the idle-sleep timer (the run didn't fail ÔÇö it *stalled* until the user woke the Mac, reading as "the appliance gave up"). `PipelineRunner` now holds a `RunSleepAssertion` and, on every `state` mutation (`didSet ÔåÆ syncSleepAssertion`), sets it held = `state.values.contains(where: \.keepsMachineAwake)`. `keepsMachineAwake` is **`.running`-only** ÔÇö `.queued` is deliberately excluded (the dequeue gap is sub-second, far below any idle-sleep timer; reviewers split on this, see the doc-comment). Design choices that are load-bearing, not incidental: **(1) `.userInitiated`, not `.idleDisplaySleepDisabled`** ÔÇö disables idle *system* sleep but lets the display dim (no reason to burn the backlight for an unwatched job; `pmset -g assertions` shows one `PreventUserIdleSystemSleep` and NO display assertion, the considerate signal a Mac power-user checks for). **(2) It does NOT prevent *forced* sleep** ÔÇö lid-close on battery sleeps the Mac regardless; that's an OS guarantee, and recovery is the resume path's job, not the assertion's. **(3) Driven off `state` (single source of truth), not instrumented per start/attach/exit site** ÔÇö `state` only mutates on coarse transitions (progress churn lives in `liveData`, not `state`), and `setHeld` is idempotent, so it can't thrash or leak. **(4) `beginActivity` is sandbox-clean** (process-local power hint, no entitlement ÔÇö unlike the `proc_listpids`/`security`/`lsof` execs that this file is otherwise scar tissue from). The `reason` string omits the app name (`pmset` already attributes it to `app.bristlenose`) and says "Analysis" not "pipeline" per the chrome convention. Edge machine + `keepsMachineAwake` mapping unit-tested in `RunSleepAssertionTests.swift` (begin/end injected, real `ProcessInfo` untouched).
- **A reconnected (orphan-attached) run reads "ResumingÔÇª", not "AnalysingÔÇª", in the row subtitle.** `PipelineProgress.attachedFromOrphan` (set once at orphan-attach, `PipelineRunner` ~`:598`) was previously *set but never read*. It now feeds `RunProgressSubtitle.compose(resuming:)` via both callers (`ProjectRow.runningSubtitle`, `SidebarSubtitleText.activityText`). `resuming` swaps the generic lead verb "Analysing" ÔåÆ "Resuming" **only in the indeterminate gap** (no known stage yet) ÔÇö the moment a real `run_progress` stage event flows, the stage verb leads ("Transcribing ┬À 7 of 8 ┬À ÔÇª") and the resume is self-evident from motion. This is the HIG-endorsed silent-inline-recovery pattern (no banner, no toast ÔÇö `absence is information`); "ResumingÔÇª" earns its keep by narrating the otherwise-suspicious quiet moment right after relaunch. Locale key `chrome.pipeline.resuming` in all 7 `desktop.json`. The `attachedFromOrphan` flag never clears, so correctness rests on callers only consulting it while `.running` (true today ÔÇö `ProjectSubtitle.resolve` owns terminal states); a fresh owned spawn writes a new `PipelineProgress` with the flag `false`. Tests: `RunProgressSubtitleTests.resuming*`.
- **Empty-space deselection in `List(selection: Set<>)` on macOS 26** ÔÇö `List(selection: $selection)` with a `Set` binding no longer auto-deselects when clicking empty space on macOS 26. SwiftUI gesture workarounds (`DragGesture(minimumDistance: 0)` as `.simultaneousGesture`) don't fire reliably. Fix: `SidebarDeselectMonitor: NSViewRepresentable` with `NSEvent.addLocalMonitorForEvents(matching: .leftMouseDown)`. In the handler, find the sidebar `NSTableView` (first table view in the window ÔÇö WKWebView detail has none), convert the click to table coordinates, call `row(at:)`. If it returns -1 (click below all rows and within the table bounds), clear `selection = []`. Always `return event` to pass through
- **AppKit `NSOutlineView` mode rows: fire click side-effects from `selectionIndexesForProposedSelection`, NOT `shouldSelectItem`; detect the click from the proposed set, NOT `clickedRow`/`NSApp.currentEvent`.** The flag-gated source-list sidebar (`ProjectSidebarOutline.swift`, `BristlenoseAppKitSidebar`) folds the 5 mode rows (lenses) into the same outline as the project list ÔÇö kept *genuinely* selected so the table draws the exact system source-list capsule, but filtered out of `SidebarSelection` (lens nodes return `nil`). Two non-obvious delegate traps, each cost a debugging session (full write-up `docs/design-desktop-sidebar-appkit.md` ┬º6; both marked DEAD END in the code): **(1) `shouldSelectItem` is NOT consulted for a row that won't end up selected.** When `selectionIndexesForProposedSelection` overrides the proposal to keep the project, the table never calls `shouldSelectItem` for the clicked lens ÔÇö so a side effect fired there (`onActivateLens`/`switchToTab`) silently never runs (zero `shouldSelect` lines in the live trace). Put activation in the proposal handler, the one delegate reliably handed the clicked lens. **(2) `clickedRow` + `NSApp.currentEvent` event-gating misfires inside the proposal handler.** At the instant it runs, `currentEvent` is not a `.leftMouseDown` on a genuine click, so an `isMouseClick` gate reads `clickedRow == -1` ÔåÆ the proposal resolves to *deselect* ÔåÆ the project is dropped ("No Project Selected"). The reliable signal is the **proposed set itself**: AppKit puts the clicked (even unselectable) row INTO `proposedSelectionIndexes` ÔÇö read the mode-click from that, no event introspection. Both lessons reduce to: **read the table's own truth; don't reconstruct user intent from `NSApp.currentEvent`.** Related: the active-lens capsule follows `activeTab` *honestly* (lags the async `switchToTab` route by one cycle) ÔÇö optimistic pre-selection flickers because `update()` re-runs `applySelection` faster than the route resolves.
- **`NSOutlineView` variable row heights need `rowSizeStyle = .custom` ÔÇö every other style silently ignores `heightOfRowByItem`.** The source-list cell (`ProjectSidebarOutline.swift`) sets per-row heights via the `heightOfRowByItem` delegate (single-line idle rows vs two-line status rows, `ProjectCellSpec.rowHeight`). But `rowSizeStyle` defaults to `.default`, and `.default`/`.small`/`.medium`/`.large` all pin a FIXED style height and **never call** the delegate ÔÇö no error, no warning, the height work is simply **inert** (rows stayed cramped ~25pt vs the native 32pt source-list pitch). Only `.custom` consults `heightOfRowByItem`. This cost a full QA round: a row-pitch "fix" was committed and did *nothing* until `.custom` was set (`ProjectSidebarOutline.swift:126`). **Rule: set `rowSizeStyle = .custom` whenever you implement `heightOfRowByItem`** ÔÇö you then own ALL row heights **and** the icon sizing (the style's automatic source-list icon sizing is gone, so set an explicit `NSImage.SymbolConfiguration(pointSize:)`; a bare `.medium`-scale config renders smaller than native, and a too-tight icon-column width then down-scales it further ÔÇö two compounding ways to get cramped icons). Height + icon size are then pixel-tuned against native in Pixelmator; the measured constants live in `ProjectCellSpec` (32pt pitch, icon pointSize 16, 20pt column ÔÇö all confirmed against Finder/Notes/Mail overlays, 22 Jun).

## Wiring menu actions (bridge handler cookbook)

### The 3-file chain

Every menu action follows the same path:

```
MenuCommands.swift                    ÔåÆ bridgeHandler.menuAction("find")
  Ôåô callAsyncJavaScript
bridge.ts                             ÔåÆ window.dispatchEvent(CustomEvent("bn:menu-action"))
  Ôåô event listener
AppLayout.tsx (or useKeyboardShortcuts) ÔåÆ React store call / DOM action
```

**Swift side is complete** ÔÇö all ~65 menu actions call `bridgeHandler.menuAction(...)`. Two frontend listeners handle them:
- **`AppLayout.tsx`** ÔÇö panel toggles, find actions, and modal/export actions (things that need AppLayout state)
- **`useKeyboardShortcuts.ts`** ÔÇö quote/player actions (things that need FocusContext/QuotesContext closures)

### Adding a new handler

1. **No Swift changes needed** ÔÇö the menu item already dispatches via `bridgeHandler.menuAction("actionName")`
2. **Choose the right listener** ÔÇö if the handler needs FocusContext/QuotesContext/PlayerContext, add it to `useKeyboardShortcuts.ts`'s `handleMenuAction` switch. Otherwise add it to `AppLayout.tsx`'s `bn:menu-action` handler
3. **Delegate to existing logic** ÔÇö most actions already have implementations in `useKeyboardShortcuts.ts` or React stores

### Action catalogue

See `docs/design-desktop-menu-actions.md` for the full catalogue (65+ actions across AppLayout, useKeyboardShortcuts, project ops, codebook stubs, edit ops), payload conventions, `getState()` stubs, and recommended implementation order.

**Quick pointers:**
- AppLayout (`bn:menu-action` handler) owns panel toggles, find actions, modals, exports, zoom, dark-mode toggle, codebook dispatches
- `useKeyboardShortcuts.ts` (`handleMenuAction` switch) owns quote/player actions that need FocusContext/QuotesContext/PlayerContext closures
- Payloads only for data the native side has that the web side doesn't (e.g. `findNext` text from `NSPasteboard.find`)

## Run / sidecar / signing gotchas (8 Jun 2026 ÔÇö full desktop-run debug)

These are the layered blockers that broke desktop ChatGPT runs end-to-end. Each hid the next; the resolution ledger (`llm_resolve` lines, below) is what finally surfaced the root.

- **`run --llm` default beats the desktop-injected provider env var.** The `run`/`analyze` CLI option `--llm` previously defaulted to `"claude"` and was *always* passed into `settings_kwargs`, so `load_settings` saw it as a CLI override ÔÇö which outranks `BRISTLENOSE_LLM_PROVIDER` (the env var the Swift host injects from `activeProvider`). Net: GUI says ChatGPT, run goes to the Anthropic endpoint, the model env var (`gpt-4o`) rides along ÔåÆ 404 `model: gpt-4o`. Fixed by defaulting `--llm` to `None` and only passing it when explicitly set (`bristlenose/cli.py`). **When adding a CLI flag whose value the desktop also injects via env, default it to None** ÔÇö a non-None CLI default silently wins over the host injection. Pinned by `tests/test_desktop_config_resolution.py::TestRunCommandDefaultDoesNotOverrideEnv`.
- **Ollama model choice rides a DIFFERENT env axis than cloud ÔÇö `BRISTLENOSE_LOCAL_MODEL`, not `BRISTLENOSE_LLM_MODEL`.** `BristlenoseSettings` has two model fields: `llm_model` (cloud, ÔåÉ `BRISTLENOSE_LLM_MODEL`) and `local_model` (Ollama, ÔåÉ `BRISTLENOSE_LOCAL_MODEL`, default `llama3.2:3b`). For the `local` provider, `llm_model` is *cosmetic* ÔÇö execution reads `local_model` (`config.py` `_fill_provider_default_model` docstring; `LLMClient` `model = self.settings.local_model`). `overlayPreferences` originally injected the resolved model into `BRISTLENOSE_LLM_MODEL` for *all* providers, so a desktop Ollama run silently dropped the user's pick (`gemma4:31b`) and fell back to the `llama3.2:3b` default ÔÇö which is too small for structured output and 100%-fails topic segmentation ("All topic segmentation calls failed", schema echoed back as the response `input_value={'$defs': ÔÇª}`). The manifest records the cosmetic `llm_model` (`gemma4:31b`) while `llm-calls.jsonl` telemetry shows the real `gen_ai.request.model=llama3.2:3b` ÔÇö that split is the diagnostic tell. Fixed (29 Jun 2026): `overlayPreferences` routes by provider ÔÇö `provider == "local"` ÔåÆ `BRISTLENOSE_LOCAL_MODEL`, else `BRISTLENOSE_LLM_MODEL` (`BristlenoseShared.swift`). Pinned by `ServeManagerEnvTests/overlayPreferences_local_provider_routes_model_to_local_axis`. **When adding any provider whose Python execution reads a non-`llm_model` field, route its env injection to the matching axis ÔÇö don't assume `BRISTLENOSE_LLM_MODEL` is universal.**
- **Bundled sidecar transcription SIGKILLs without JIT entitlements.** The run path transcribes via `mlx-whisper ÔåÆ numba ÔåÆ llvmlite`, which JIT-compiles word-timestamp kernels and `mprotect`s pages executable. Under Hardened Runtime that's killed with `EXC_BAD_ACCESS / SIGKILL (Code Signature Invalid)` / `CODESIGNING Invalid Page` ÔÇö surfaces in the desktop as "Run failed (no terminus event) / unknown" with **no output dir created** (crash before any stage writes). Fix: `com.apple.security.cs.allow-jit` + `com.apple.security.cs.allow-unsigned-executable-memory` in `desktop/bristlenose-sidecar.entitlements` (re-sign after). Diagnose via `~/Library/Logs/DiagnosticReports/bristlenose-sidecar-*.ips`. **App-Store-review-sensitive** ÔÇö route through app-store-police before submit. The serve-only entitlement comment ("JIT not needed") was scoped to serve; run needs it.
- **PyInstaller sidecar needs `multiprocessing.freeze_support()`.** A dep (mlx-whisper/tokenizers/onnxruntime) spawns mp children by re-executing the frozen binary with interpreter bootstrap args (`-B -S -c ÔÇª`); without `freeze_support()` in `desktop/sidecar_entry.py:main()` those children fall through the argv rewrite, become `serve -B`, and Typer kills them ("No such option '-B'" / `resource_tracker: process died unexpectedly`). Main pipeline survives but child workers crash-loop. `freeze_support()` runs the worker and exits; no-op for a normal invocation. Must run before any argv manipulation.
- **The app stores LLM keys in keychain-access-group `app.bristlenose` ÔÇö CLI `security` can't read them.** Keys set in the sandboxed desktop app go into the app's keychain group (`KeychainHelper.swift`), invisible to `security find-generic-password`. Only keys set via the CLI `bristlenose configure` (plain generic passwords, no access group) are CLI-readable. So you cannot recover the app's OpenAI key from the terminal; for a CLI repro of a desktop run, pass `BRISTLENOSE_<PROVIDER>_API_KEY` explicitly. (Under the sandbox the Swift host reads the group key and injects it; Python never touches Keychain.)
- **Sidecar build provenance ÔÇö the banner SHA carries `-dirty`/`@buildtime`.** `bristlenose/_build.py` resolves a build SHA: generated `_build_info.py` baked by `build-sidecar.sh` (bundle) ÔåÆ live `git rev-parse` (dev, with `-dirty`) ÔåÆ `"unknown"`. The banner (`_print_header`, also captured into `last-run-failure.log`) shows e.g. `e93e079 @06-08T02:46Z` or `e93e079-dirty`. Use it to confirm a rebuild actually took ÔÇö a bare SHA with no `@time`/`-dirty` distinction was the flaw that made "did my rebuild land?" unanswerable.
- **A stale bundled sidecar silently no-ops any new *Python-emit* feature on desktop ÔÇö QA of such features starts with a sidecar rebuild.** The bundled sidecar is built from a point-in-time `.venv-sidecar` and is NOT rebuilt by Xcode (see the build-sidecar note above). So a feature whose Python side emits new data ÔÇö e.g. the Phase 0b `run_progress` events that feed the determinate ring *and* the sidebar progress text (`RunProgressSubtitle`) ÔÇö looks completely dead on desktop (ring stuck as a spinner, subtitle stuck at "AnalysingÔÇª") while the *same* `bristlenose run` from a current venv works. The events file is the tell: it carries only the older lifecycle events (`run_started` / `run_completed`) and **zero `run_progress`**. **Diagnose by comparing the run's `run_started.process.bristlenose_version` (in `<output>/.bristlenose/pipeline-events.jsonl`) against the tree `__version__`** ÔÇö a mismatch means the bundled sidecar predates the emit code. Fix: `desktop/scripts/build-sidecar.sh` then `SIGN_IDENTITY=- desktop/scripts/sign-sidecar.sh`, then Cmd+R (Copy Sidecar Resources re-bundles it). The whole 0b ring shipped *latent* on desktop for ~1 day this way ÔÇö the Swift rendered correctly but the bundled sidecar (v0.15.14) predated `RunProgressEvent`; nobody saw the ring or text until the 18 Jun 2026 rebuild. (Also under sandbox=on, projects.json + events live in `~/Library/Containers/app.bristlenose/Data/ÔÇª`, not `~/Library/Application Support/` ÔÇö look in the container when inspecting a sandboxed run's state.)
- **Resolution ledger for "why this provider/model?".** `load_settings` builds a step-by-step `llm_resolve` ledger (inputs ÔåÆ alias ÔåÆ pydantic-winner+source ÔåÆ orphan-guard ÔåÆ final), replayed from `LLMClient.__init__` and the early-logging hook in `run()` (config + preflight run *before* `pipeline._configure_logging`, so live emission is otherwise lost). `llm_preflight_validate` logs the exact wire pair where a mismatch 404 fires. Grep the run log (`<output>/.bristlenose/bristlenose.log`) for `llm_resolve` / `llm_preflight_validate` / `llm_client_target` / `llm_call_failed` before theorising about provider bugs.

## Testing

### Framework choice

- **Swift Testing** (`@Test`, `#expect`) for all unit and integration tests
- **XCTest** (`XCTestCase`, `XCUIApplication`) reserved for UI tests only ÔÇö it's the only framework with UI automation
- Both can coexist in the same test target, but **never mix assertions** ÔÇö `#expect` inside an `XCTestCase` is silently ignored, and `XCTFail` inside a `@Test` doesn't register ([interop proposal ST-0021](https://forums.swift.org/t/st-0021-targeted-interoperability-between-swift-testing-and-xctest/84965) is in progress)

### Conventions

- **If a SwiftUI view is making a decision, the decision belongs in a testable helper, not in the view.** Drop hit-testing, menu enable state, error categorisation, availability mapping ÔÇö pull each into a plain `struct` or free function that takes inputs and returns an outcome (e.g. `DropDecision.evaluate(items:target:) -> DropOutcome`, `RevealAvailability.from(project:) -> Bool`, `LocateValidation.classify(folder:) -> LocateResult`, `ProjectSubtitle.resolve(...) -> SubtitleVariant` ÔÇö the sidebar subtitle precedence chain, lifted out of `ProjectRow`'s body). The view becomes a thin caller; the helper gets a unit test in `BristlenoseTests/`. This is the minimum-step convention before reaching for an XCUITest target ÔÇö covers the bulk of Swift-layer testability without new infrastructure. See `docs/design-test-philosophy.md` for the philosophy
- Test file naming: `{ClassName}Tests.swift` in `BristlenoseTests/`
- **Suite-level `@MainActor` for tests that touch `@MainActor` types** ÔÇö annotate the `@Suite struct`, not each `@Test func`. The per-method approach (`@MainActor @Test func ...`) is noisier and tends to miss tests that call static methods on actor-isolated types. Two existing files (`I18nTests`, `ProjectIndexTests`) had header comments saying *"X is @MainActor ÔÇö all tests must be @MainActor too"* but were missing the suite-level annotation; adding it cleared all the actor-isolation errors at once. Swift Testing runs `@Test` on arbitrary executors by default
- KeychainHelper tests always use `InMemoryKeychain`, never real SecItem ÔÇö avoids overwriting real API keys (SIGKILL bypasses teardown, so cleanup is not crash-safe)
- ProjectIndex tests always use `ProjectIndex(fileURL: tempURL)`, never the default Application Support path
- I18n tests use `configure(localesDirectory:)` with fixtures in `BristlenoseTests/Fixtures/`, never `findLocalesDirectory()` (which hardcodes dev paths)

### Test target setup

**`BristlenoseTests` is wired into the Xcode project** (`xcodebuild -list` shows both `Bristlenose` and `BristlenoseTests` targets). Run it with:

```sh
cd desktop/Bristlenose
xcodebuild build-for-testing -scheme Bristlenose -configuration Debug -destination 'platform=macOS,arch=arm64'
xcodebuild test-without-building -scheme Bristlenose -destination 'platform=macOS,arch=arm64' -only-testing:BristlenoseTests
```

Note `test-without-building` runs the **already-compiled** bundle ÔÇö after editing a test file you must re-run `build-for-testing` first, or the stale bundle runs and your change appears to have no effect.

The target uses `PBXFileSystemSynchronizedRootGroup`, so new `.swift` files added to `BristlenoseTests/` are auto-discovered ÔÇö no pbxproj edit needed per file.

**Auto-sync flattens flat-resource folders into `Resources/` and collides** ÔÇö e.g. `Fixtures/{en,es}/*.json` (locale-style folders that aren't `.lproj`) all copy to `Resources/<name>.json` with no per-locale subdir, causing `Multiple commands produce '...common.json'` build errors. Fix in `Bristlenose.xcodeproj/project.pbxproj`: add a `PBXFileSystemSynchronizedBuildFileExceptionSet` referencing the synced root with `membershipExceptions = (Fixtures/en/common.json, ...)` listing each colliding file. Folder paths (`Fixtures` or `Fixtures/en`) don't work as exceptions ÔÇö must list each file explicitly. The proper Xcode-side fix (folder reference instead of group) preserves the subdir structure but requires the GUI; pbxproj exceptions are the CLI-friendly workaround. Same exception set can also exclude pre-existing test files that don't yet compile under Swift 6 strict mode

Build settings must match the app target:
- `SWIFT_VERSION` ÔÇö same as app (currently 5.0)
- `SWIFT_DEFAULT_ACTOR_ISOLATION = nonisolated`
- `ENABLE_APP_SANDBOX = NO` (today; matches app target; flips to YES for alpha)

### Running tests

```bash
xcodebuild test -project desktop/Bristlenose/Bristlenose.xcodeproj -scheme Bristlenose -destination 'platform=macOS'
```

Or in Xcode: Cmd+U.

**Concurrent `xcodebuild test` runs across worktrees hang in teardown ÔÇö read the per-test "passed" lines, not the exit code.** When two worktrees run `xcodebuild test-without-building` for the same `Bristlenose` scheme at the same time (e.g. two Claude sessions, this branch + a sibling like `warm-sidecar-pool`), they contend on the shared macOS test host and **wedge in post-execution teardown** ÔÇö the tests themselves run and pass, but `xcodebuild` never returns. A `timeout`-wrapped run then exits 124, and a manual `kill` of the wedged process shows up as exit 144 (128+SIGTERM) ÔÇö **both look like test failures but aren't.** Diagnose by reading the log: `grep -c "Test case.*passed"` and `grep -cE "Ô£ÿ|recorded an issue|Testing failed"` (zero failures + "Testing started completed" = green, regardless of the wrapper exit code). It clears on its own once the sibling run finishes (then a clean run returns exit 0). Hit 19 Jun 2026 on `project-status-line` while `warm-sidecar-pool` was testing concurrently.

### Testability refactors

Two injection points exist for safe testing:

1. **`ProjectIndex(fileURL:)`** ÔÇö pass a temp directory URL to avoid touching `~/Library/Application Support/Bristlenose/projects.json`
2. **`KeychainStore` protocol** ÔÇö `KeychainHelper.liveStore` for production, `InMemoryKeychain()` for tests. The static `KeychainHelper.get/set/delete` methods remain unchanged for existing call sites
