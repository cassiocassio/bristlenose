› How the score works — and the prior art it borrows from

Two numbers, side by side rather than blended — because a single figure hides the failure mode the field warns about: a component can be high-coverage but risky, or low-coverage but harmless.

The unit — a “themeable declaration”

Only declarations whose property should answer to the design system count toward the denominator: colour / background / border-colour, font family·size·weight·line-height, spacing (margin·padding·gap), radius, box-shadow, z-index, transition/duration. Structural geometry (display, position, flex, grid, layout width/height) is excluded. A small ignore-set is never a violation: 0, transparent, currentColor, inherit, 1px hairlines. Each themeable declaration is one of inside (a --bn-* token) · fixable (a literal that already equals an existing token) · outside (a literal with no token — missing family, off-scale, magic z-index, dangling ref).

Coverage — the metric you asked for

Coverage = (inside + 0.5 × fixable) ÷ themeable declarations × 100 100% = every themeable value is a token · 0% = every value is a raw literal

The industry’s de-facto signal: stylelint-declaration-strict-value flags each declaration where a token-worthy property carries a raw literal, so Coverage is just 1 − (violations ÷ targeted declarations), with half-credit for fixable drift — non-compliant today, but a one-line change with no decision to make.

Health — the borrowed refinement

Coverage treats a hardcoded brand colour that breaks dark mode the same as an off-scale margin. Health doesn’t. Following SonarQube’s per-rule-weight → normalise → band model, each violation carries a severity:

weightwhatwhy it hurts
3missing-token / hardcoded colour, dangling refbreaks theming and dark mode
1off-scale spacing·radius·duration, magic z-index, shadow geometrycosmetic drift, no theming risk
1fixable drifttrivial, but still off-system
Health = (1 − Σ(weight × violations) ÷ (3 × themeable declarations)) × 100

So blockquote and timecode — clean except for off-scale spacing — score moderate Coverage but high Health, while badge (missing danger/success tokens, hardcoded tints) sinks on both. That gap is the whole point: it tells you where the effort should go.

Roll-up & honest limits

Per-declaration → per-component → per-lens, weighted equally. On reproducibility: the per-component tallies below are a curated hand audit (so each carries an annotated defect + fix), but the metric is now computable from source — ✓ scripts/audit-css.py parses the token values and scans every component file, printing per-file and per-lens Coverage/Health with the same formula (it reads live CSS, so it already reflects the danger/success + z-index fixes). Its whole-theme read is 81% coverage / 91% health across 55 files — a broader, machine-checkable baseline than these five content lenses. Remaining rough edge: components are weighted equally, whereas the adoption literature weights by render frequency (a workhorse should count more than a one-off) — the obvious next iteration.
